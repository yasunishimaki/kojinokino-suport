# 就労ナビAI「ココマデ」デモアプリ設計書 v1.0

作成日: 2026年7月6日
対象: フェーズ1(音声＋手動報告版)のデモンストレーション用アプリ

---

## 1. 目的とスコープ

### 1.1 このデモアプリの目的

提案先(就労支援機関・企業・当事者)に対して、以下の3つの中核体験を実際に触って理解してもらうこと。

1. **進捗の照会** — 「今どこまでやった?」に即答する体験
2. **次の一歩の提示** — 手順書と照合して「次は○○です(出典付き)」と案内する体験
3. **意図の再生** — 席を立つ前の一言宣言を、あとから本人の言葉のまま返す体験

### 1.2 スコープ(含むもの / 含まないもの)

| 含む(フェーズ1相当) | 含まない(フェーズ2以降) |
|---|---|
| 手順書の登録・照合 | スマートグラス連携 |
| プッシュ型の問いかけ(標準所要時間ベース) | カメラによる作業自動認識 |
| 意図宣言の記録・再生 | PCログの自動収集(デモでは手動/擬似) |
| 出典提示・「記録がありません」応答 | 複数利用者の管理 |
| 音声入出力 | 商用レベルの認証・セキュリティ |

デモの本質は「記録からのみ抽出する照合型AI」の信頼感を体感させることにある。豪華さより、応答の正確さと出典表示を優先する。

## 2. デモシナリオ(プレゼンで見せる流れ)

所要約5分。プレゼンター1名で完結できる構成。

1. **[支援員視点]** 手順書登録画面で、その日の作業「請求書データ入力(5工程)」を登録する。各工程に標準所要時間を設定(例: 工程2は15分)
2. **[本人視点]** ナビ画面に「現在: 工程1 伝票の仕分け / 次: 工程2 データ入力」が表示される
3. 工程1完了を報告(ボタンまたは音声「終わりました」)→ ナビが工程2へ進む
4. **[プッシュ型問いかけ]** デモ用に短縮した標準所要時間が経過 → アプリから音声で「工程2のデータ入力は終わりましたか?」と問いかけ → 「まだ」と答えると「了解しました。続けてください」
5. **[意図の宣言]** 離席ボタンを押すと「一言どうぞ」→ 「見積書を印刷してきます」と吹き込む
6. 戻ってきて「あれ、何しようとしてたっけ?」と質問 → 「3分前に『見積書を印刷してきます』と言っていました(意図ログ 14:02)」と本人の言葉＋出典で応答
7. **[信頼性の実演]** わざと記録にないこと(「明日の予定は?」)を質問 → 「記録がありません」と答えることを見せる ← **ここがデモの山場。生成AIなのに知ったかぶりをしない、を目の前で証明する**

## 3. 画面構成

### 3.1 本人用ナビ画面(メイン画面)

認知負荷を最小化する設計。一画面・一情報が原則。

```
+--------------------------------------+
|  現在の作業                           |
|  ■ 工程2: データ入力                  |   ← 最大表示・常時
|  (B列 15行目まで完了)                 |   ← 進捗サブ表示
|--------------------------------------|
|  次: 工程3 印刷・確認                 |   ← 一段小さく
|--------------------------------------|
|  [ 終わった ] [ 今どこ? ] [ 離席する ] |   ← ボタンは3つまで
|--------------------------------------|
|  🎤 音声でも話しかけられます           |
+--------------------------------------+
```

- 文字は大きく(本文20pt以上)、色は高コントラスト、アニメーションなし
- 音声応答は「一度に1情報・短文」。長い応答は「続きを聞きますか?」で分割
- 問いかけ(プッシュ通知)はチャイム音+音声+画面表示の3チャネル同時

### 3.2 支援員用画面

- **手順書登録**: 作業名 / 工程リスト(名称・説明・標準所要時間・完了の目印) / 対象者
- **ダッシュボード**: 当日の進捗ログ・意図ログ・問いかけへの応答履歴を時系列表示
- **設定**: 問いかけの猶予時間(標準所要時間の何%超過で聞くか)、音声速度、繰り返し回数

### 3.3 意図宣言モーダル

離席ボタン押下または「行ってきます」等の発話で起動。

```
「一言どうぞ」 → 🎤録音(最大10秒) → テキスト化して確認表示 → [OK] で意図ログに時刻付き保存
```

## 4. データ構造

Supabase(PostgreSQL)またはローカルJSONで保持。デモはローカルでも可。

```
procedures(手順書)
  id, work_name(作業名), created_by(支援員),
  steps: [ { step_no, title, description,
             standard_minutes(標準所要時間),
             completion_sign(完了の目印: 例「保存ボタンを押した」) } ]

progress_log(進捗ログ)
  id, procedure_id, step_no, status(started/done/paused),
  detail(例: "B列15行目"), reported_by(self/system_prompted),
  timestamp, source_ref(出典: どの報告・操作から記録されたか)

intent_log(意図ログ)
  id, spoken_text(本人の発話そのまま), transcribed_text,
  timestamp, context(直前の作業step_no), resolved(再生済みフラグ)

qa_log(問いかけ・応答ログ)
  id, question, answer, answered(true/false), timestamp
```

設計原則: **すべてのレコードに timestamp と source_ref を持たせる。** 出典提示はこの2つから機械的に生成する。

## 5. 中核ロジック

### 5.1 プッシュ型問いかけ(標準所要時間トリガー)

```
工程開始時刻 + 標準所要時間 × 猶予係数(初期値1.2) を経過
  → まだ完了報告がなければ問いかけを発火
  → 「工程2のデータ入力は終わりましたか?」(はい/まだ/ヘルプ の3択)
  → 「はい」→ progress_log に done を記録(reported_by: system_prompted)
  → 「まだ」→ タイマーを50%延長して再セット
  → 無応答×2回 → 支援員ダッシュボードに通知
```

### 5.2 照合型応答(記録からのみ抽出するルールの実装)

Anthropic APIを使用。**システムプロンプトで応答を記録に限定する。**

```
あなたは高次脳機能障害のある方の作業を支えるナビゲーターです。
以下のルールを絶対に守ってください。

1. 回答は、以下に渡す「手順書」「進捗ログ」「意図ログ」の内容からのみ行う
2. 記録にない質問には、必ず「記録がありません」とだけ答える。推測・一般知識での補完は禁止
3. すべての回答の末尾に出典を付ける。形式: (進捗ログ 14:02 / 手順書 工程3)
4. 一度に伝える情報は1つ。2文以内。専門用語は使わない
5. 意図ログを再生するときは、本人の発話をそのまま引用する

[手順書] {procedures をJSONで挿入}
[進捗ログ] {当日の progress_log}
[意図ログ] {当日の intent_log}
```

応答後、アプリ側でも検証を行う(二重チェック): 応答文に出典タグが含まれない場合は表示せず「記録がありません」に差し替える。

### 5.3 中断検知(デモ版)

フェーズ1デモでは以下の2方式を実装。

- 手動: 「離席する」ボタン → 意図宣言モーダル起動
- 擬似自動: 一定時間(デモでは30秒)操作がない → 「作業を中断していますか? 一言残しますか?」

将来のPCログ連携(ウィンドウ切替検知)・カメラ検知は、この中断イベントの発火元を差し替えるだけで済む構造にしておく(イベント駆動設計)。

## 6. 技術構成(デモ版)

| 層 | 技術 | 備考 |
|---|---|---|
| フロントエンド | React(単一ページ) | タブレット・PC両対応のレスポンシブ |
| 音声入力 | Web Speech API(SpeechRecognition) | 日本語認識。Chrome推奨 |
| 音声出力 | Web Speech API(SpeechSynthesis) | 速度調整可能に |
| AI応答 | Anthropic API(claude-sonnet-4-6) | 5.2のシステムプロンプトで照合限定 |
| データ | ローカルstate(デモ) → Supabase(実証時) | デモはリロードで初期化されてよい |
| タイマー | フロントエンドで管理 | デモ用に「時間を早送り」ボタンを用意 |

デモ用の仕込み: プレゼン中に15分待てないため、標準所要時間を秒単位に短縮できる「デモモード」スイッチを設定画面に置く。

---

## 7. 認知アクセシビリティ要件(本人向けUI)

- 一画面につき伝える情報は1つ。選択肢は最大3つ
- 音声は「短く・ゆっくり・繰り返し可能」。再生ボタンでいつでも聞き直せる
- 問いかけは責める調子にしない。「終わりましたか?」であって「まだですか?」ではない
- 意図宣言は失敗できない設計: 録音が聞き取れなくても「録音は保存しました」と伝え、音声そのものを再生できるようにする(テキスト化失敗時のフォールバック)
- 色だけに意味を持たせない(色覚多様性対応)。アイコン＋文字を併用

## 8. フェーズ2/3への拡張ポイント

| 現在の部品 | フェーズ2(表示グラス) | フェーズ3(カメラAIグラス) |
|---|---|---|
| ナビ画面 | グラス片眼表示に要約版を送出 | 同左 |
| 完了報告 | ボタン/音声(変更なし) | 映像認識→「終わったように見えます。合っていますか?」確認後に記録 |
| 中断検知 | PCログ(ウィンドウ切替)連携 | 離席・来客・口頭指示の自動検知 |
| データ構造 | 変更なし(共通データ基盤) | source_refに「映像由来(本人確認済み)」区分を追加のみ |

**データ構造とシステムプロンプトの3原則(記録限定・出典・確認)は全フェーズ不変。** これが段階導入モデルの技術的裏付けとなる。

## 9. 開発ステップ(目安)

1. モック画面(本人用ナビ+意図宣言)を作り、操作フローを確認 — まずここまででも提案先に見せられる
2. 手順書登録とタイマー問いかけを実装(AIなしで動くコア)
3. Anthropic API接続と照合プロンプトの調整(「記録がありません」の確実性テスト)
4. 音声入出力の組み込み
5. デモシナリオ(第2章)の通し練習・デモモード調整
